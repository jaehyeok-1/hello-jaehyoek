# KBO 우취(경기 취소) 정보 웹사이트 구축 가이드

이 저장소는 **"KBO 10년 경기 + 기상 데이터"**를 모아서,
사용자가 `날짜 + 구장`을 입력하면 우취 가능성을 보여주는 서비스를 만들기 위한 최소 골격입니다.

## 1) 목표 기능

- 과거 10년 KBO 전 경기 데이터 수집
  - 경기일, 구장, 시작시간, 경기 여부(정상/우천취소/기타 취소)
- 같은 시각/구장의 기상 데이터 결합
  - 강수량, 강수형태, 습도, 풍속, 기온, 시간당 변화
- 예측/판단
  - "비가 왔지만 경기 진행" 케이스까지 포함해 확률형으로 판단
- 웹 UI
  - 사용자 입력: 월/일, 구장(또는 정확한 날짜), 필요시 경기시작시간
  - 출력: 우취 확률 + 근거 변수 + 유사 과거 경기 사례

---

## 2) 아키텍처(권장)

1. **수집(ETL)**
   - KBO 경기 일정/결과 소스 수집
   - 기상청(관측/예보) API 수집
2. **정제/매핑**
   - 구장 ↔ 기상 관측소 매핑 테이블 관리
   - 날짜/시간 기준으로 경기와 날씨 조인
3. **학습/평가**
   - 이진 분류: `우취 여부` (1/0)
   - 시간 분할 검증(과거→미래)
4. **서빙 API**
   - 입력값 기반 feature 생성 후 모델 추론
5. **프론트엔드**
   - 예측 결과 + 설명(왜 이런 판단인지)

---

## 3) 데이터 모델

`sql/schema.sql`에 기본 테이블 정의를 넣었습니다.

- `games`: KBO 경기 원천 정보
- `weather_hourly`: 시간 단위 관측/예보
- `stadium_station_map`: 구장-관측소 매핑
- `game_weather_features`: 모델 입력 feature 저장

## 4) 핵심 설계 포인트

### A. "비가 왔는데 경기한 날"을 반드시 학습에 포함

우취는 단순히 "강수 여부"가 아니라,
- 강수 **강도/지속시간/경기시간대 일치**
- 구장 배수/관리 상태
- 구단 및 리그 운영 의사결정
같은 요소의 결합 결과입니다.

그래서 타겟은 `우취(1) / 정상진행(0)`로 두고,
"강수 있음 + 정상진행" 사례를 **음성 클래스(0)**로 포함해야 모델이 현실적으로 동작합니다.

### B. 누출(leakage) 방지

- 경기 종료 후에만 알 수 있는 변수(예: 최종 누적 강수)를 학습 입력에 넣지 않기
- 실제 의사결정 시점(예: 경기 2~3시간 전) 기준으로 feature를 만들기

### C. 평가 방식

- 랜덤 split보다 **시계열 split** 권장
- 지표: ROC-AUC + PR-AUC + threshold별 precision/recall
- 실서비스는 확률값과 함께 근거를 제시

## 5) 빠른 시작

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

API 예시:
- `GET /health`
- `POST /predict` (날짜/구장 입력)

- 모델 버전/학습일자/훈련 데이터 기간을 기록
